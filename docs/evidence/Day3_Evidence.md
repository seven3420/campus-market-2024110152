# Day3 Evidence - Mock 数据建模与列表渲染

## 1. 今日完成内容

Day3 的核心任务是将项目从 Day2 的静态页面骨架推进到数据驱动阶段。今日完成了以下工作：

1. **Mock 数据设计**：在 `db.json` 中为二手交易、失物招领、拼单搭子、跑腿委托四类业务设计了数据结构，每类业务包含 6 条贴近校园生活的数据。
2. **JSON Server 配置**：在 `package.json` 中配置了 `mock` 启动命令，使用 JSON Server 在 `http://localhost:3001` 提供模拟 API 服务。
3. **Axios 请求封装**：创建 `src/api/http.ts` 作为 Axios 基础实例，并分别为四类业务创建了独立的 API 模块（`trade.ts`、`lostFound.ts`、`groupBuy.ts`、`errand.ts`），每个模块导出 TypeScript 接口类型和请求方法。
4. **通用组件开发**：创建了 `ItemCard.vue` 通用列表卡片组件和 `EmptyState.vue` 空状态组件，四个业务页面统一使用这些组件进行列表渲染。
5. **页面数据渲染**：在 `TradeView.vue`、`LostFoundView.vue`、`GroupBuyView.vue`、`ErrandView.vue` 四个核心页面中，通过 `onMounted` 生命周期请求接口数据并渲染列表。

## 2. Mock 数据结构说明

| 数据集合 | 对应业务 | 主要字段 | 页面用途 |
|---|---|---|---|
| trades | 二手交易 | title、price、category、condition、location、publisher、publishTime、status | 展示二手商品列表，突出价格和成色 |
| lostFounds | 失物招领 | title、type、itemName、location、eventTime、contact、status | 区分寻物/招领，展示物品信息和联系方式 |
| groupBuys | 拼单搭子 | title、type、targetCount、currentCount、deadline、location、publisher | 展示拼单进度和截止时间 |
| errands | 跑腿委托 | title、taskType、reward、from、to、deadline、publisher、status | 展示任务类型、酬劳和取送路线 |

**数据状态设计**：
- `open`：进行中（在售/待认领/拼单中/待接单）
- `closed`：已结束（已售/已认领/已结束）
- `done`：已完成（仅跑腿委托使用）

## 3. 我的设计

在动手编写 Mock 数据之前，我先分析了四类校园业务的核心特征：

**二手交易为什么需要 `price` 和 `condition`？**
价格是二手交易最核心的信息，直接决定浏览者是否有兴趣点击。成色（condition）则是二手商品区别于全新商品的关键属性——九成新和七成新的定价策略完全不同，买家需要通过成色判断性价比。如果缺少这两个字段，列表就会退化成普通的信息流，失去了"交易"属性。

**失物招领为什么需要 `type` 字段？**
失物招领包含两个截然不同的行为：丢失物品的人发布寻物启事（lost），拾到物品的人发布招领启事（found）。用 `type` 字段区分这两种场景，页面可以据此展示不同的标签样式（如"寻物"用橙色、"招领"用绿色），用户也能一眼区分。

**拼单搭子为什么需要 `targetCount` 和 `currentCount`？**
拼单的核心逻辑是"凑够人"，目标人数和当前人数是判断拼单能否成功的关键指标。这也是拼单与普通团购的本质区别——拼单有截止时间和人数门槛，用户需要看到进度才能决定是否参与。缺少进度字段，用户无法判断一个拼单是否还"差人"。

**跑腿委托为什么需要 `from`、`to` 和 `reward`？**
跑腿的本质是"从 A 点到 B 点的代办服务"。`from` 和 `to` 描述任务路线，让接单者评估是否顺路；`reward`（酬劳）是接单者的直接动力。这三者共同构成了跑腿委托的核心信息三角。我特意将报酬字段命名为 `reward` 而非 `price`，因为跑腿是劳务报酬而非商品价格，语义上更准确。

## 4. AI 设计

在数据建模过程中，我使用 AI 辅助工具参与了数据结构设计。

**AI 帮助生成的内容**：
- 初步的 `db.json` 数据结构骨架
- 四类 API 模块的基础代码框架
- `ItemCard.vue` 和 `EmptyState.vue` 组件的基础模板
- 页面中 `onMounted` 请求数据的基本模式

**AI 设计的优点**：
- AI 建议为每类数据统一使用 title、description、location、status 等核心字段，保证了四类数据的基础结构一致性，便于后续组件复用
- AI 建议使用 TypeScript interface 为每类数据定义类型，这让字段引用有了编译期检查，减少了字段名拼写错误
- AI 建议将 Axios 实例和业务 API 分文件管理，这种分层让代码可维护性更好

**AI 设计中我发现的问题**：
- AI 初版在二手交易中加入了 `originalPrice`（原价）和 `viewCount`（浏览量），这些字段对校园轻集市场景来说过于复杂——学生之间的二手交易更关注现价和成色，原价参考意义有限，浏览量则属于后续优化时才需要的指标。我选择删除这两个字段，保持数据结构简洁
- AI 初版将跑腿委托的报酬字段命名为 `price`，这与二手交易的 `price` 混淆。我将其改为 `reward`，因为"跑腿费"和"商品价格"在语义上是不同的
- AI 一度生成了用户表、评论嵌套等关联结构，这些是 Day5/Day6 才需要考虑的内容，Day3 阶段只需要扁平的数据列表，我全部删除了
- AI 在失物招领数据中遗漏了 `type` 和 `itemName` 字段，我手动补充了这两个字段——`type` 用于区分遗失和招领，`itemName` 用于快速标识物品类型

## 5. 最终调整

在实际开发过程中，我对初始设计做了以下调整：

1. **字段重命名**：将跑腿委托的 `price` 改为 `reward`，将 `category` 改为 `taskType`，将 `initiator` 统一改为 `publisher`，让字段名更贴合各自业务语义
2. **删除冗余字段**：删除了二手交易的 `originalPrice` 和 `viewCount`，删除了拼单中的 `price` 和 `originalPrice`（拼单重点是人数组团而非商品定价），删除了失物招领中的 `category`（用 `type` + `itemName` 替代更精准）
3. **补充缺失字段**：为失物招领补充了 `type`（lost/found）和 `itemName` 字段，让寻物和招领两种场景在同一个列表中清晰区分
4. **统一时间字段命名**：二手交易使用 `publishTime`（发布时间），失物招领使用 `eventTime`（事件发生时间），拼单和跑腿使用 `deadline`（截止时间）——不同类型使用不同时间语义，比统一用 `createTime` 更准确
5. **简化组件设计**：ItemCard 保留 title/description/tag/location/time 五个核心 props，通过 footer 插槽让各页面按需注入价格、进度、状态等差异化内容，避免 props 膨胀
6. **数据内容贴近校园**：所有 Mock 数据都使用真实的校园场景（宿舍楼、食堂、图书馆、教学楼），价格控制在学生可接受范围（2-850元），发布人使用"小王""研二学长"等校园化称呼

## 6. 遇到的问题与解决方法

**问题：JSON Server 端口被占用**
在启动 `npm run mock` 时遇到 `EADDRINUSE` 错误，端口 3001 已被之前的 JSON Server 进程占用。
**解决方法**：使用 `lsof -i :3001` 查找占用进程并 `kill`，然后重新启动。同时意识到在多次开发过程中需要注意后台进程管理。

**问题：TypeScript 类型导入报错**
更新 API 接口类型名后（如 `Trade` → `TradeItem`），编辑器提示"模块没有导出的成员"。
**解决方法**：这是 IDE 诊断缓存导致的延迟警告，实际执行 `vue-tsc --build` 类型检查时已通过，确认导入名称与导出名称一致即可。清理 IDE 缓存后问题消失。

**问题：字段映射不一致导致数据展示异常**
初版中 `db.json` 的字段名（如 `createTime`）与 API 接口定义（`publishTime`）不一致，导致页面渲染时时间字段显示为空。
**解决方法**：以 `db.json` 为单一数据源，确保 API 接口类型定义与 JSON 数据字段完全一致。模板中引用的字段名必须同时匹配接口类型和 JSON 数据。

## 7. 今日反思

Mock 数据建模、JSON Server 接口模拟和列表渲染这三个环节，本质上是前端开发中"业务理解 → 数据抽象 → 接口对接 → 页面呈现"这条完整链路的缩影。

Mock 数据不是随便编几条测试内容，它是对业务对象的结构化表达——字段的设计反映了开发者对业务的理解深度。比如跑腿委托中 `from` 和 `to` 两个字段，看似简单，背后是对"跑腿本质是点到点服务"这一业务特征的理解。

JSON Server 的价值在于提供了一个轻量级的接口模拟环境，让前端开发可以在不依赖真实后端的情况下验证数据请求链路。它让"从静态页面到数据驱动页面"的过渡变得可控——先确认数据结构合理，再确认接口通畅，最后确认页面渲染正确。

列表渲染是数据驱动页面的起点。通过 Axios 请求数据、赋值给响应式变量、在模板中用 `v-for` 渲染列表，这个流程几乎是所有前端业务页面的基本模式。在 Day3 搭建好这个模式后，后续的发布表单、状态管理、详情页开发都会基于这套数据流进行扩展。

总的来说，Day3 的任务量看似不大，但它在整个实训中起到了"承上启下"的关键作用：向上承接了 Day2 的页面骨架，向下为 Day4-D7 的交互逻辑和状态管理奠定了数据基础。
