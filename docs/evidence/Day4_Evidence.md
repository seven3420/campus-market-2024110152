## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 二手商品需要展示分类、价格和成色，帮助买家快速判断商品价值 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分"寻物"和"招领"两种场景，物品名称和发生时间是关键检索条件 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 拼单需要明确目标人数和截止时间，让其他人判断是否值得参与 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 跑腿任务需要明确取件地、送达地和酬劳，这是接单者最关心的三个要素 |

所有类型共用 title、location、description 三个通用字段，因为它们在任何发布场景下都是必要的元信息。

## 3. 表单校验规则

| 校验项 | 适用类型 | 规则 | 理由 |
|---|---|---|---|
| 标题 | 全部 | 不能为空 | 标题是列表展示的核心信息，必须有内容 |
| 地点 | 全部 | 不能为空 | 所有校园信息都需要明确地点 |
| 描述 | 全部 | 不能为空 | 详细描述帮助信息接收方做出判断 |
| 商品分类 | 二手交易 | 不能为空 | 分类是筛选和查找的核心维度 |
| 价格 | 二手交易 | 必须大于 0 | 二手商品需要明确的售价 |
| 成色 | 二手交易 | 必须选择 | 成色直接影响商品价值和购买意愿 |
| 物品名称 | 失物招领 | 不能为空 | 物品名称是失物招领的核心检索字段 |
| 发生时间 | 失物招领 | 不能为空 | 时间信息对失物找回至关重要 |
| 拼单类型 | 拼单搭子 | 不能为空 | 类型帮助用户筛选感兴趣的拼单 |
| 目标人数 | 拼单搭子 | 不能少于 2 人 | 拼单至少需要 2 人参与才有意义 |
| 截止时间 | 拼单/跑腿 | 不能为空 | 时效性是拼单和跑腿的关键约束 |
| 任务类型 | 跑腿委托 | 不能为空 | 任务类型是筛选和匹配的基础 |
| 酬劳 | 跑腿委托 | 不能为负数 | 酬劳是接单的动力，不应为负 |
| 取件地点 | 跑腿委托 | 不能为空 | 明确从哪里开始任务 |
| 送达地点 | 跑腿委托 | 不能为空 | 明确任务的终点 |

## 5. 人工调整内容

在 AI 生成代码的基础上，我进行了以下人工调整：

1. **保留项目现有样式变量**：PublishView.vue 的样式使用了项目 main.css 中定义的 CSS 自定义属性（如 `--color-primary`、`--color-card`、`--radius-lg`），保持与项目整体风格一致
2. **使用 Omit 类型**：API 的 create 函数参数使用 `Omit<TradeItem, 'id'> & { id?: number }` 类型，既允许不传 id（让 JSON Server 自动生成），又保持与已有接口类型的一致性
3. **添加焦点样式**：为输入框添加了 focus 状态的边框高亮和外发光效果，提升交互体验
4. **添加 onTypeChange**：切换发布类型时自动清除错误提示，避免上一种类型的校验错误残留在新类型表单上
5. **保留 Element Plus 的设计令牌兼容**：虽然表单本身不使用 Element Plus 组件，但样式变量保持与项目全局一致

## 7. 遇到的问题与解决方法

### 问题 1：发布类型切换后错误提示残留
- **现象**：在"二手交易"中提交失败（有校验错误），切换到"失物招领"后之前的错误提示仍然显示
- **解决方法**：在 `onTypeChange` 函数中调用 `clearErrors()`，切换类型时清空所有错误信息

### 问题 2：API 接口类型兼容
- **现象**：现有 `TradeItem` 接口中 `id` 是必填的 `number`，但新增数据时不应手动指定 id
- **解决方法**：使用 `Omit<TradeItem, 'id'> & { id?: number }` 类型，将 id 设为可选，由 JSON Server 自动管理

## 8. 今日反思

发布表单、表单校验和数据新增是 Web 应用从"只读展示"走向"用户交互"的三个关键环节。发布表单决定了用户能够向系统输入哪些信息——字段设计是否贴合业务、是否简洁明了，直接影响用户是否愿意使用这个功能。表单校验是数据质量的第一道防线——在前端进行基础校验可以减少无效请求、提升用户体验，同时也让后端接口收到的数据更加规范。数据新增则是将用户输入转化为持久化业务数据的关键步骤——每一次成功的 POST 请求都在丰富平台的内容生态。

在校园轻集市这个场景中，四类业务（二手交易、失物招领、拼单搭子、跑腿委托）的字段虽然有重叠，但各自有不同的关键信息。设计时不能简单用一套字段覆盖所有场景，也不能为每种类型做完全独立的表单——在通用性和差异化之间找到平衡，才是合理的表单设计。通过 Day4 的实践，我理解了"用户输入 → 前端校验 → 接口提交 → 反馈跳转"这一完整数据流，为后续的用户状态管理、我的发布记录等功能打下了数据基础。
