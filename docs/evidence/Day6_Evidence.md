# Day6 Evidence - 交互优化与体验完善

## 1. 今日完成内容

1. 为各列表页添加了空状态提示（使用 EmptyState 组件）；

## 2. 交互优化详细说明

### 2.1 空状态处理

在 `TradeView.vue`（二手交易）、`UserCenterView.vue`（个人中心收藏列表）中集成了 `EmptyState` 组件：
- 列表无数据时展示引导提示文字
- 避免空白页面给用户造成困惑

### 2.2 加载与错误状态

加载和错误状态目前以轻量方式集成在各页面中：
- 列表页在 `onMounted` 中使用 try/catch 捕获请求异常，通过 `console.error` 记录错误
- 发布页在 `handleSubmit` 的 catch 块中通过 `window.alert` 提示用户检查 Mock 服务
- 未使用独立的 LoadingState/ErrorState 组件（权衡：当前数据量小、加载快，独立组件收益有限）

### 2.3 收藏交互优化

- 收藏按钮使用 `@click.stop` 阻止事件冒泡，避免点击收藏时触发卡片跳转
- 按钮文案和样式在"收藏"/"已收藏"之间切换，提供即时视觉反馈
- 个人中心提供"取消收藏"按钮，完成收藏的增删闭环

### 2.4 表单交互优化

- 提交按钮在 `submitting` 状态下显示"提交中..."并禁用，防止重复提交
- 切换发布类型时自动清空校验错误提示
- 重置按钮清空所有表单字段和错误信息

## 3. 关于独立状态组件（LoadingState / ErrorState / SearchBar）的说明

Day6 推荐创建的独立 `LoadingState.vue`、`ErrorState.vue` 和 `SearchBar.vue` 组件未在当前项目中作为独立组件实现。原因如下：

- **LoadingState**: 当前数据量较小（Mock 本地服务），加载速度极快，独立的加载组件收益不明显。各页面通过数据响应式（列表为空时展示 EmptyState）已覆盖基本体验。
- **ErrorState**: 错误处理采用轻量方式（控制台日志 + alert），适合当前项目阶段。如果后续接入真实后端，可以引入独立的重试/错误展示组件。
- **SearchBar**: 搜索功能目前通过 JSON Server 的 `?q=` 查询参数在部分页面内联实现，未抽取为独立组件。后续可统一抽取以提升复用性。

这些是当前阶段基于"不过度工程化"原则做出的取舍。

## 5. 遇到的问题与解决方法

**问题 1: 收藏按钮点击触发页面跳转**

点击 ItemCard 上的收藏按钮时，由于卡片本身有点击事件跳转详情页，点击收藏按钮也会触发跳转。

**解决方法**: 在收藏按钮上添加 `@click.stop` 阻止事件冒泡。

**问题 2: AI 建议引入过多优化方案**

AI 在交互优化阶段倾向于推荐 loading 动画库、全局 toast 通知、骨架屏、虚拟滚动等方案。

**处理方式**: 人工评估后，选择与当前项目规模和阶段匹配的方案——保持轻量，不过度工程化。拒绝了全局拦截器和第三方 toast 库的建议。

## 7. 今日反思

Day6 的交互优化工作让我意识到：前端开发不只是"把功能做出来"，还需要关注用户在使用过程中的体验细节。空状态的引导文案、按钮的即时反馈、重复提交的防止——这些看似微小的工作，直接影响了用户对产品的感受。

同时，交互优化的边界也很重要。AI 工具倾向于推荐各种"最佳实践"——骨架屏、全局错误拦截、虚拟滚动、性能监控等。但并非所有优化都适合当前项目阶段。判断"什么需要做"和"什么不需要做"，需要开发者对项目目标、用户场景和技术成本有自己的判断。这也是 AI 辅助开发中的一项关键能力。
