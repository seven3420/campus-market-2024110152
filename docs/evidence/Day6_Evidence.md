## 3. 问题修复记录

- **列表页请求失败时页面空白**：原来四个业务列表页在 catch 中只写了 `console.error`，没有设置错误状态变量，导致请求失败时页面完全空白。现在增加了 `error` 响应式变量，模板中使用 `ErrorState` 组件展示错误信息和重试按钮。
- **加载状态缺失**：原来各页面没有 loading 状态，用户打开页面时如果数据加载较慢会看到空白或旧数据残留。现在增加了 `loading` 响应式变量和 `LoadingState` 组件，请求前显示加载动画。
- **搜索结果为空无提示**：原来没有搜索功能，现在增加了搜索后，如果关键词过滤结果为空，会通过 `EmptyState` 组件显示"暂无符合条件的 XX 信息"。
- **状态判断顺序修正**：模板中严格按照 `v-if="loading"` → `v-else-if="error"` → `v-else-if="filteredItems.length === 0"` → `v-else` 的顺序排列，确保错误状态优先于空状态展示。
- **收藏按钮状态不明显**：原来收藏和未收藏的按钮样式几乎相同，难以区分。现在增加了 `.favorite-btn.active` 样式（蓝色背景 + 蓝色文字），点击后按钮从"🤍 收藏"变为"❤️ 已收藏"并高亮。
- **删除无用文件**：移除 ListView.vue（含硬编码静态测试数据，未注册路由）和 ProfileView.vue（仅写了"个人中心页面将在后续 Day 中完善"的占位页，实际已有 UserCenterView.vue）。

## 7. 遇到的问题与解决方法

**问题 1：loading 状态一直不消失**
- 原因：最初只在 try 块中设置 `loading.value = false`，catch 分支忘记设置，导致请求失败时页面一直显示加载动画
- 解决：使用 `try/catch/finally` 结构，在 finally 中统一设置 `loading.value = false`，确保无论成功或失败都关闭加载状态

**问题 2：搜索框输入中文时列表过滤不及时**
- 原因：使用 `@input` 事件配合 `v-model` 在中文输入法下可能触发不及时
- 解决：SearchBar 组件使用 `:value` + `@input` 的手动绑定方式，配合 `emit('update:modelValue')` 确保每次输入都触发更新。同时在父组件中使用 `computed` 计算过滤结果，保证响应式更新

**问题 3：错误状态和空状态判断顺序错误**
- 原因：最初将 `EmptyState` 放在 `ErrorState` 前面判断，导致请求失败且列表为空时显示了空状态而非错误提示
- 解决：严格按照 loading → error → empty → 正常列表的顺序排列模板条件判断，错误状态优先级高于空状态

## 8. 今日反思

一个前端项目不能只关注"功能能跑"，还需要关注交互反馈、错误提示和用户体验，原因在于：真实用户在使用过程中会遇到各种边界情况——网络延迟导致加载等待、Mock 服务未启动导致请求失败、搜索关键词无匹配结果、重复点击按钮导致重复提交等。如果项目只在"一切正常"的情况下能跑，一旦出现异常，用户面对的就是空白页面或静默失败，完全不知道发生了什么、该如何处理。

通过今天的交互优化，我理解了三种核心状态的必要性：**加载状态**让用户知道系统正在工作而非卡死；**错误状态**告诉用户出了什么问题以及如何恢复；空状态让用户明白"没有数据"是一个确定的结果而非 bug。此外，搜索筛选让信息查找更高效，按钮的状态反馈让操作结果一目了然。

Day6 是从"功能完成"走向"可用、易用、可展示"的关键一步。体验优化不是美化页面，而是让用户在任何情况下都能理解当前状态、知道下一步该做什么。这对于 Day7 的最终展示尤为重要——一个状态反馈完整、交互体验流畅的项目，远比一个功能堆砌但到处是空白和静默失败的项目更具展示价值。
