## 4. AI 生成内容的问题

在 7 天开发过程中，AI 生成内容出现过以下典型问题：

| 问题类型 | 具体表现 | 出现阶段 |
|----------|----------|----------|
| 字段不贴合业务 | AI 生成的 Mock 数据使用英文通用字段名，与校园中文场景不符 | Day3 |
| 代码过度复杂 | 建议引入骨架屏、全局错误拦截器、toast 通知库等 | Day6 |
| 超出范围的功能 | 自动补充登录注册、图片上传、支付、聊天等未要求的功能 | Day4, Day6 |
| 组件抽象过度 | 建议将简单表单字段也抽取为独立组件 | Day4 |
| 生成内容无法直接运行 | 部分 TypeScript 类型推断与项目实际配置存在细微差异 | Day5 |
| README 虚构功能 | 在 README 中自动添加"支持微信支付、实时聊天"等描述 | Day7 |

## 5. 我的判断与修改

在 7 天实训中，我对 AI 生成内容的判断和修改主要遵循以下原则：

1. **业务贴合度优先**: AI 生成的是通用前端模式，需要人工将其转化为校园生活场景的具体表达。例如 Mock 数据中的字段名、示例文案、业务类型标签等，都需要替换为校园场景用语。

2. **复杂度控制**: AI 倾向于给出"工业级"方案，但课程项目有其学习目标和复杂度边界。我多次拒绝了 AI 建议的过度抽象和第三方库引入，保持项目在可控复杂度内。

3. **功能边界判断**: AI 会不自觉地在表单中补充"图片上传"、在 README 中添加"支付功能"等。每次 AI 输出后，我都会逐项核对是否在任务要求范围内，删除超出部分。

4. **代码风格统一**: AI 在不同阶段生成的代码风格可能不一致（import 路径写法、组件命名风格等），需要人工统一。

5. **状态边界决策**: Day5 中哪些数据放入 Store、哪些保留在组件内，这个架构决策需要自己对页面间数据流有理解，AI 只能提供代码实现，不能代替架构判断。

## 6. 总结

AI 在本项目中显著提高了**重复性编码工作**的效率——页面骨架搭建、Mock 数据生成、API 封装模板、表单字段渲染等机械性工作，AI 可以快速生成基本可用的代码，节省了大量手写样板代码的时间。

但以下部分**仍然必须由开发者自己理解和判断**：

- **业务理解**: AI 不理解"校园轻集市"的业务场景，需要开发者将通用模式转化为具体业务表达；
- **架构决策**: 哪些数据入 Store、组件如何拆分、状态边界在哪里，这些需要开发者对应用结构有整体把握；
- **复杂度权衡**: 引入什么、不引入什么，什么阶段做什么事，AI 不会考虑学习目标和项目边界；
- **代码审查**: AI 生成的代码需要逐行审查，修正类型错误、统一风格、删除多余内容；
- **功能真实性**: AI 可能虚构功能，开发者需要确保文档和代码描述一致、真实。

通过 7 天实训，我最大的收获不是"AI 能帮我写多少代码"，而是**学会了如何管理 AI、审查 AI、修正 AI，并保持自己对项目的掌控力**。这种能力——判断 AI 的输出、做出取舍、保证最终质量——可能比手写代码本身更重要。
